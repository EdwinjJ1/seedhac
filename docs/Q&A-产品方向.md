# Q&A · 产品方向

> 模拟评委提问 + 我们的回应。用来对齐团队思路、打磨决赛话术。

---

## 评委挑战

### 挑战 1 · 你们做的是「飞书内的 ChatGPT 包装器」，不是 Agent

> 「群聊 → @Agent → 出文档 + PPT + 任务」

这套流程今天用豆包 + 复制粘贴，3 分钟手动也能跑完。你们的 Agent 没有自主决策——它只是被 @ 一下、按固定 pipeline 跑一遍。

赛题写得很清楚：「**Agent 思维融入产品设计，AI 驱动核心流程，而非简单堆砌 GUI**」——你们这套，把 GUI 换成了 IM 卡片，本质是一回事。

---

### 挑战 2 · 「多端」是你们最弱的一环，但赛题最看这个

赛题原文：「**应对移动端与桌面端差异，保证多端体验的一致性与流畅性**」——这不是配菜，这是赛道命脉。

你们方案里的「多端」全靠飞书 App 自带的多端能力——你们没做任何端侧工作，只是写了一个服务端 webhook。

评委会追问：

- 你们的 Agent 在地铁里弱网用户 @ 一下，会发生什么？
- 桌面端正在编辑的文档，移动端能看到光标位置吗？
- 移动端键盘弹起遮住卡片按钮，你们怎么处理？

---

### 挑战 3 · 「蒸馏」是个伪命题——飞书妙记已经做了

飞书妙记今年早就上了：**会议自动转写 + AI 摘要 + 行动项提取 + 一键生成飞书文档**。

你们的「会议蒸馏」和妙记功能集 80% 重叠，差异只在两处：

1. 你们处理的是群聊文字而不是会议语音 → 但群聊本来就是文字，不需要「蒸馏」，需要的是「组织」
2. 你们能生成 PPT → 这是唯一亮点，但只占整个 pipeline 的 20%

**致命问题**：你们的产品上线第一天，用户会问「这和妙记 + 一键生成 PPT 有什么不一样？」 你们准备怎么回答？

如果答案是「集成度更高」——那这是个 feature，不是产品。

## 我们的回应

### 回应挑战 1 —— 我们不是「被 @ 才动的工具」

实际流程是：**人在前台聊，AI 在后台干活，人想用的时候东西已经在桌上了**。

它不是「被 @ 才动的工具」，而是「在你说话时已经在跑的同事」。

```
传统：用户 → 命令 → Agent 干活 → 出结果

我们：用户 → 聊天（没命令）
            ↓（Agent 在后台监听）
            检测信息缺口
            ↓
            主动检索 / 查文档 / 找历史
            ↓
            在你想要的那一刻，结果已经摆在那
```

→ **挑战 1 解了**。

---

### 回应挑战 3 —— 我们解决的是真痛点：对话中的信息差

两个人聊着聊着发现「那个数据是多少来着」「上次那个客户叫什么」「我们之前是不是定过这事」——全部得停下来去搜。

我们的 Agent 把「停下来去搜」这一步消灭了。

这是**真省时间**的产品，不是「看起来很 AI」的产品。

跟妙记拉开距离：**妙记是事后总结，我们是事中预判**。完全不同的时间维度，**不重叠**。

→ **挑战 3 解了**。

---

### 回应挑战 2 —— 多端的故事自然成立

- **手机端**：Agent 推送一张「我帮你查到了」卡片，点一下就能分享
- **桌面端**：边聊边在侧栏看到 Agent 的实时调研

这两端**必须配合才完整**——手机端负责即时触达，桌面端负责深度查看。

→ **挑战 2 自动解了**。

## 这个想法的两个新问题

### 问题 1 · 怎么判断「信息缺口」？

- 两个人聊「咱们周五吃啥」—— Agent 要不要主动搜餐厅？
- 两个人聊「那个 Q3 数据」—— Agent 要不要去翻报表？

判断标准是什么？**不能每句话都触发，那是骚扰；也不能等用户喊，那就是回到老路**。

**可能的解法**（4 选 1 或组合）：

- **不确定性触发**：检测到「那个…」「上次…」「是多少来着」等模糊表述
- **任务型触发**：检测到讨论里出现「决定 / 方案 / 计划」等词
- **疑问触发**：有人提了一个**别人没回答的问题**
- **历史关联触发**：当前对话和过去 N 天某个文档 / 讨论高度相关

> 这个判断逻辑就是我们整个产品的「灵魂」—— **做好了是神，做差了就是骚扰机器人**。

---

### 问题 2 · 成本控制

每条消息都过一次 LLM 判断要不要搜——**这个费用扛不住**，比赛 demo 跑两天可能就把 API quota 烧光。

**可能的解法**：

- **两层模型**：便宜的小模型做「是否触发」判断（甚至本地正则 / 关键词），贵的大模型只在触发后跑
- **滑窗批处理**：每 N 条消息或每 30 秒攒一批，不每条都判
- **冷启动**：群第一次开通时跑一次「群上下文画像」，后续判断成本骤降

> 这是**工程深度**，评委爱听。

## 总结 · 重新包装后的产品定位

> **「飞书 Sentinel」—— 一个住在群聊里的隐形同事**
>
> 它不打断你说话，但你说着说着发现需要某个数据、某段历史、某份文档时——
> **它已经在群边上把答案摆好了**，配一张可以一键分享、一键继续追问的卡片。
>
> 它做对话中的「信息预取」，让你**永远不必停下来去查**。

这就是我们去 pitch 的一句话。
