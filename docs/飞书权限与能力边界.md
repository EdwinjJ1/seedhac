# 飞书权限与能力边界

> 决定我们能做什么 / 不能做什么的「平台天花板」文档
>
> 调研日期：2026-04-27 · 维护人：Evan
>
> 用途：开发前 sanity check + 决赛话术里「为什么我们这样设计」的依据

---

## TL;DR（一页能看完）

| 维度 | 结论 |
|---|---|
| 一个企业能装多少个机器人 | **不限**，每个自建应用都可以开启「机器人」能力。我们项目里「每个员工配一个 agent」=「在同一个企业管理后台下创建 N 个自建应用」，技术上可行；运营上飞书企业管理员要批 N 次权限。 |
| 机器人能否监听员工 A 与员工 B 的私聊 | **不能**。`im:message.p2p_msg` 只能监听**用户与机器人之间**的单聊；员工之间的 1v1 私聊飞书 API 不开放，企业管理员后台也没有合规出口。 |
| 机器人能否监听群消息 | **能**，前提是机器人在群里。`im:message.group_msg` 接收所在群所有消息；`im:message.group_at_msg` 只接收 @ 机器人的消息。 |
| 机器人能否主动给某个员工发私聊 | **能**，但只能发给「曾经和它建立过单聊会话」或「企业内部已授权的」用户；用 `im:message:send_as_bot` + `receive_id_type=open_id`。**首次主动私聊：用户必须先在客户端「添加」过这个机器人**（点过头像、@过它，或被管理员推送）。 |
| 前端能改到什么程度 | **5 个入口，自由度递增**：①消息卡片（JSON 拼组件）②机器人菜单（私聊输入框上方按钮）③网页应用（完整 HTML/JS，飞书 webview 加载，PC/移动通用）④小程序⑤多维表格插件。**做不到的**：在 IM 主界面注入 sidebar、改输入框行为、监听草稿。 |
| 流式输出 | **支持**。Card 2.0 + CardKit OpenAPI 提供 streaming update，0.5s 节流，万字长文本可用。 |
| 妙记 | API 能拿到「音视频下载链接」「文字记录」「妙记元信息」「统计数据」；不能直接订阅「妙记生成」事件，也不能让机器人主动发起妙记。 |
| 多维表格 | API 完整：增删改查 + 字段 + 视图 + 高级权限。**10 QPS** + 单批 ≤ 500 条。可以当我们的「Memory + 知识图谱」存储。 |
| Aily | 企业级 Agent 平台，OpenAPI 仍是 beta，自建应用可调；它和我们的方案是**竞品**，不是基础设施。我们用底层 IM/Bitable API 自建，不走 Aily。 |
| API 频控 | 单租户单机器人 **100 次/分 + 5 次/秒**；超限返回 429，响应头里有 `x-ogw-ratelimit-reset`。 |

## 一、机器人（Bot）

### 1.1 两种机器人，差别极大

| 类型 | 谁能建 | 能力 | 我们用哪个 |
|---|---|---|---|
| **自定义机器人**（群机器人） | 任何群主 | 只能在所在群里**单向**收 webhook → 发消息。不能响应 @、不能交互。 | ❌ 不用 |
| **应用机器人**（自建应用 → 开机器人能力） | 企业开发者 | 完整 OpenAPI：双向收发、卡片回调、订阅事件、调通讯录/Bitable/妙记 | ✅ 全用这个 |

> 「Sentinel」/「员工专属 agent」必须走应用机器人。下文「机器人」一律指应用机器人。

### 1.2 一个企业能否有多个机器人

可以。**「一个机器人」= 「一个自建应用 + 开启了机器人能力」**。同一企业管理员可以创建多个自建应用，互相独立的 App ID / Secret / 权限范围 / 事件订阅。

> 这直接回答了袁晨杰那条「每个员工有专属 agent」的问题：技术路径就是给每个员工创建（或在 metadata 里挂）一个 app instance。**MVP 期一个机器人共用即可**——多 app 主要是隔离权限和合规边界，比赛 demo 不需要。

### 1.3 关键限制（合规天花板）

- ❌ **不能监听员工 A 与员工 B 的 1v1 私聊**——飞书 API 没有这个 scope。即使企业管理员也没有「读员工私聊」的合规出口（这是飞书产品级红线，不是技术限制）。
- ❌ **不能监听机器人不在的群**——必须先被拉进群，事件才会推过来。
- ⚠️ **首次主动私聊有门槛**——用户没和机器人有过任何交互前，机器人 `send_as_bot` → 该 user_id 会失败或被静默丢弃；解决方式是企业管理员把机器人「推送给指定员工」（管理后台有这个动作）或员工自己点过机器人头像建立会话。
- ⚠️ **外部群 ≠ 内部群**——外部群（跨企业）需要单独打开「Bot supports external groups」配置，权限矩阵更窄。

### 1.4 我们要勾的权限清单（一次性勾全，免得后期反复审）

```
# 消息收发
im:message                          # 收发消息（基础）
im:message:send_as_bot              # 以机器人身份发消息
im:message.group_at_msg             # 接收群里 @ 机器人的消息
im:message.group_msg                # 接收群里全部消息（关键！我们要监听非 @ 消息）
im:message.p2p_msg                  # 接收用户主动私聊机器人的消息

# 群组
im:chat                             # 读写群信息
im:chat:readonly                    # 读群基础信息
im:chat.group_info                  # 读群成员、群描述等

# 通讯录（最小集，识别说话人是谁）
contact:user.base:readonly          # 用户名、头像
contact:user.employee:readonly      # 工号、部门（识别"老板 vs 员工"用）
contact:department.base:readonly    # 部门信息

# 多维表格（Memory 存储）
bitable:app                         # 读写多维表格

# 妙记
minutes:minutes:readonly            # 读妙记文字记录 + 元信息
# 注意：「下载妙记的音视频文件」这个权限正在替换旧的「导出妙记文件」

# 云文档（推送 PPT/Doc 用）
docx:document                       # 读写文档
wiki:wiki:readonly                  # 读 Wiki
```

> 这一套是 demo 全功能集，权限批量导入 JSON 飞书后台支持。

## 二、IM 消息（事件订阅）

### 2.1 我们用「长连接」而不是「webhook」

`@larksuiteoapi/node-sdk` 的 `WSClient` 起一条长连接，**免公网 / 免域名 / 免回调签名**。事件直接推到本地 process，开发期最爽。

代价：长连接断开后事件会重传一段；不适合生产高可用，但决赛 demo 完全够用。

### 2.2 关键事件

| 事件 | 触发时机 | 用在哪 |
|---|---|---|
| `im.message.receive_v1` | 群里收到任何消息（含 @ 机器人 / 非 @）| Sentinel 监听 → 缺口检测 |
| `card.action.trigger` | 用户点了卡片按钮 / 提交了表单 | 「一键分享」「我不需要这个」回调 |
| `p2p_chat_create_v1` | 用户首次私聊机器人 | 触发「冷启动 onboarding」 |
| `im.chat.member.bot.added_v1` | 机器人被拉进群 | 触发「群上下文画像」预跑 |

### 2.3 「全量监听群消息」是合法的吗

合法，但前提是：

1. 机器人**已经被群主/管理员拉进群**——用户在群里看得到机器人的存在；
2. 应用申请了 `im:message.group_msg` 并通过审核（自建应用在测试企业里**无需审核**，发布出去要审）；
3. 群成员可以在群设置里看到「机器人能读消息」的提示——这是飞书前端的合规提示，不需要我们额外做。

> **不是后台静默监听**。从员工视角看，群里有机器人 = 机器人能看到大家说啥，这是飞书的产品契约。

### 2.4 历史消息

`GET /open-apis/im/v1/messages?container_id_type=chat&container_id=...` 拉群历史。

- 需要 `im:message` + `im:chat:readonly`
- 分页 `page_size + page_token`
- **机器人必须是群成员**才能拉

## 三、UI 扩展点全景（之前漏的，重要）

> 飞书并不是「只有消息卡片能改」，而是开放了 **5 个进入客户端的入口**。每个入口能改的程度不一样。MVP 只用①+②，决赛 demo 加分项可以做④。

### 3.0 五个 UI 入口对比

| 入口 | 改到什么程度 | 写什么代码 | 装在哪 | 我们用不用 |
|---|---|---|---|---|
| ① **消息卡片** | JSON 描述 + 飞书原生组件 | 后端拼 JSON | IM 群聊 / 私聊气泡里 | ✅ 主战场 |
| ② **机器人菜单** | 几个固定按钮 + 跳链/触发事件 | 开发者后台配置 | 私聊机器人输入框上方 | ✅ 加分项 |
| ③ **网页应用 / H5** | **完整 HTML/CSS/JS**，自由度最高 | 自己写前端 + JSSDK | 工作台 / 群 Tab / 卡片跳转 | 🔶 可选 |
| ④ **小程序（gadget）** | 接近原生的体验，能调摄像头/通知 | tt.* JSAPI，包 ≤ 10MB | 工作台 / 群 Tab | ❌ MVP 不做 |
| ⑤ **多维表格插件 / 文档块** | 嵌在表格/文档里的自定义视图 | 前端 + 飞书 SDK | Bitable 侧边栏 / Doc 块内 | ❌ 跑题 |

### 3.1 入口①：消息卡片（前端主战场）

卡片用 JSON 描述（v1）或 schema-based JSON（v2 / Card 2.0），飞书客户端按规则渲染。

**不能做**：
- ❌ 自定义 HTML / CSS / JS（除非走入口③）
- ❌ 改 IM 列表样式 / 群头像 / 输入框
- ❌ 在 IM 主界面注入侧边栏

**能做**：用预置组件拼 UI（见 3.2）。

### 3.1.5 入口②：机器人菜单（被严重低估的入口）

机器人有一个**「自定义菜单」**，私聊机器人时显示在输入框上方。可配：
- 折叠样式 / 常驻样式
- 每个按钮可做三件事：①跳指定 URL（含 H5 应用）②推事件给后端 ③触发飞书内置动作
- 完全在开发者后台配置，**不写代码**

> Sentinel 可以这么用：菜单挂「📊 今日缺口报告」「🧠 我的 Memory」「⚙️ 静默 1 小时」三个按钮，比让用户记 `/command` 体验好十倍。这是「让 agent 看起来像产品」的关键。

### 3.1.6 入口③：网页应用（要不要走这条）

**这是飞书 UI 自由度的天花板**：

- 你写一个完整的 H5 网站，发布成「自建应用」
- 用户在飞书工作台点开，**飞书内嵌一个 webview 直接加载你的页面**，PC/移动端通用
- 页面里通过 JSSDK 调 `tt.*` API：免登拿用户身份、读剪贴板、调摄像头、发飞书消息……
- HTML / CSS / JS 完全你写——Tailwind / React / 任何东西都行

**用法 1（轻）**：卡片里的某个按钮点开 → 跳到我们的 H5 页面 → 展示「Memory 全图谱」「知识图谱可视化」这种卡片塞不下的复杂 UI。

**用法 2（重）**：把整个 Sentinel 控制台做成 H5 应用，挂在工作台。员工点头像就进。

**对我们的取舍**：MVP 不做（多一倍前端工作量），但**决赛 demo 演到 Memory 知识图谱时可以加这一页**——评委看到「点一下卡片 → 跳出来一个真正的图可视化」会很惊艳。

### 3.1.7 入口④⑤：小程序 / Bitable 插件（MVP 不做）

- **小程序**：飞书自有 DSL（tt.* 体系，类似微信小程序），开发链路重，包 ≤10MB，要装开发者工具调试。我们的场景纯 IM 交互，没必要走小程序。
- **多维表格插件 / 文档块**：能在 Bitable 侧边栏或 Doc 内部塞自定义 UI，但用户得先打开那张表/那篇文档才能看到。我们的入口是「群里」，方向不对。

### 3.1.8 真正不能做的事

下面这些不管走哪个入口都做不到：

- ❌ 在飞书 IM 主聊天列表里加 sidebar / 注入面板
- ❌ 改飞书原生输入框的行为（拦截、补全、AI 续写）
- ❌ 监听用户没发出去的草稿
- ❌ 在群消息上覆盖我们自己的浮层
- ❌ 做飞书启动屏 / 通知样式定制（这是企业管理员后台的能力，不是开发者的）

> 决赛话术里如果有人问「为啥不做 sidebar 实时面板」——直接答「飞书客户端没开放这个扩展点，我们用消息卡片 + 机器人菜单 + H5 应用三层补足」，比硬解释好。

### 3.2 Card 2.0 组件清单

| 组件 | 类别 | 能干啥 |
|---|---|---|
| `button` | 交互 | 点击触发 `card.action.trigger`，可带 `value` 透传业务参数 |
| `input` | 交互 | 单行输入；`max_length` 本地校验 |
| `form` | 容器 | 包裹多个 input/select，统一一次提交（避免每次输入都回调） |
| `select` | 交互 | 下拉单选 |
| `multi_select` | 交互 | 下拉多选 |
| `date_picker` / `picker_time` / `picker_datetime` | 交互 | 日期/时间 |
| `checker` | 交互 | 勾选项（待办列表） |
| `markdown` | 展示 | 富文本 |
| `div` | 容器 | 文本块 + 可附带 `extra`（按钮/图片） |
| `note` | 展示 | 灰色小字辅助说明 |
| `hr` | 展示 | 分隔线 |
| `img` | 展示 | 图片（要先用 image API 上传拿 image_key）|
| `chart` | 展示 | 飞书原生图表（柱/折/饼）|
| `column_set` | 布局 | 多列布局 |
| `action` | 容器 | 一组按钮 |
| `interactive_container` | 交互 | 整块卡片可点击 |

> Sentinel 决赛 demo 的卡片用：`markdown` + `column_set` + 三个 `button` + 一个 `note` 就够了。

### 3.3 流式输出

Card 2.0 支持 **streaming update**（CardKit OpenAPI）。一张卡片可以反复 `patch`：
- 节流推荐 0.5s/次
- 单卡片可承载万字级长文本
- 比 `message.patch`（编辑消息）更稳，专为 AI 流式打字机设计

> 这是我们「思考中... → 找到了！」体验的关键能力。

## 四、妙记（Minutes）

### 4.1 能拿到什么

| API | 用途 | 权限 |
|---|---|---|
| `GET /open-apis/minutes/v1/minutes/:minute_token` | 妙记元信息（标题、时长、所有者、封面）| `minutes:minutes:readonly` |
| `GET /open-apis/minutes/v1/minutes/:minute_token/media` | 拿音视频下载链接（24h 有效）| `minutes:minutes:readonly` 或 「下载妙记音视频文件」|
| `GET /open-apis/minutes/v1/minutes/:minute_token/transcript` | 文字记录（带时间戳、说话人）| `minutes:minutes:readonly` |
| `GET /open-apis/minutes/v1/minutes/:minute_token/statistics` | 统计数据（参与人、时长分布）| `minutes:minutes:readonly` |

### 4.2 限制

- ❌ **没有「妙记生成完成」事件**——必须轮询或由用户/老板手动把 `minute_token` 推给机器人；
- ❌ **机器人不能主动发起会议录制**——录制是用户在会议中点的；
- ⚠️ **基础版企业有妙记额度限制**，企业版/商业版无限；
- ⚠️ **`minute_token` 从妙记 URL 末段截取**，长度恰好 24 字符。

### 4.3 我们怎么用

「会议结束 → 老板把妙记链接 @ 机器人 → 机器人拉文字记录 → 进 Memory → 下次群里聊到相关话题时主动召回」。

**这是 Sentinel 与妙记的差异化**：妙记是「事后总结」，我们把妙记数据**喂回事中对话**，做信息预取。

## 五、多维表格（Bitable）

### 5.1 是什么

飞书云文档下的「带 schema 的表格 + 内置 UI」。完全可以当「Memory 数据库 + 简易知识图谱」用——这就是我们 `技术栈与可复用能力.md` 第四节的方案。

### 5.2 API 边界

| 维度 | 上限 |
|---|---|
| 单表行数 | 100 万行（新版） |
| 字段类型 | 文本/单选/多选/数字/日期/复选/附件/人员/**双向关联**/**公式**/**Lookup** |
| QPS | **10 QPS**（单 app）|
| 单次批量 | ≤ 500 条 / 次（增/删/改）|
| 事务性 | 批量操作**全成功或全失败**，无部分成功 |
| 权限 | `bitable:app`（读写）/ `bitable:app:readonly`（只读）|
| 应用接入 | 机器人要先被加为表格协作者（owner / admin / 编辑者）|

### 5.3 双向关联字段 ≈ 轻量图谱

`决策表` 一行 → 关联到 `数据表` 多行 + `负责人表` 一行。这就是我们 demo 里给评委演示的「知识图谱」：1-2 跳关系飞书 UI 直接画出来，不用上图数据库。

### 5.4 协作者陷阱

机器人 ≠ 自动是表格协作者。要么：
1. 用 user_access_token（绑定到管理员账号）先拉机器人进表；
2. 或者表格创建者手动「添加文档应用」把机器人加进去。

> 这一步是部署期的 footgun，第一次跑必踩。

## 六、Aily（飞书智能伙伴平台）

### 6.1 它是什么

**飞书官方的 Agent 创建平台**。提供：技能编排、知识库、模型路由、对话调试、一键发布到 IM/服务台/Web。

### 6.2 我们和它的关系

**它是竞品**——评委如果懂 Aily，会问「你为啥不用 Aily 直接搭一个」。

**回答**：
1. Aily 是「装配工厂」，给的是**被 @ 才动的模式**；Sentinel 要的是「在你说话时已经在跑」，必须自己写监听层；
2. Aily 的技能编排走的是有限状态机式 workflow，我们要的是 LLM 驱动的 Skill Router，自由度更高；
3. Aily OpenAPI 还在 beta，封装不深，反而比裸用 IM API 更受限。

### 6.3 自建应用能否调 Aily OpenAPI

可以。需要在自建应用里申请 `aily:*` 系列权限（`aily:file:read` / `aily:file:write` 等），然后调 `https://aily.feishu.cn` 域下的 OpenAPI。MVP 不引入。

## 七、API 频控（Rate Limit）

| 维度 | 限制 |
|---|---|
| 单租户单机器人 | **100 req/分 + 5 req/秒** |
| 自定义机器人（webhook） | 同上 |
| 多维表格 | 10 QPS（独立计） |
| 超限响应 | HTTP 429，响应头 `x-ogw-ratelimit-limit` / `x-ogw-ratelimit-reset` |

**对我们的影响**：
- 群里 1 秒来 5 条消息，每条都过小模型 → 5 req/秒 单是「写卡片回执」就到顶；
- 必须**批处理**（30s 或 10 条一批，沛彤 issue #5 的方案直接命中）；
- 流式更新卡片 0.5s 节流，吻合 5 req/秒。

## 八、合规与法律边界

| 关注点 | 结论 |
|---|---|
| 「监听员工聊天」合规吗 | ✅ **群里**合规（机器人在群里 = 群成员可见 = 知情同意）；❌ **员工之间私聊**飞书根本不开放，无需我们自检 |
| 数据要不要落到企业外 | 🔶 取决于我们把 Memory 写哪里。**写飞书多维表格 = 数据从未离开飞书**，最强合规；写本地 SQLite/Chroma = 出企业边界 |
| 豆包 token 调用算不算数据外传 | ⚠️ 算。决赛话术里提一句「Memory 全在飞书内，仅 LLM 推理时把上下文片段送豆包，且豆包是企业合作模型」 |
| 评委挑战「监听全群消息有没有 PII」 | 我们的回答：①机器人在群里全员可见；②Memory 默认只存「决策/缺口/待办」，不存原始消息；③多维表格高级权限可按部门配读取 |

## 九、申请流程（Evan 已跑通：30 + 10 + 30 分钟）

1. **30 分钟**：飞书开通账号 → 创建测试企业 → 开发者后台建「企业自建应用」→ 拿 App ID + Secret
2. **10 分钟**：勾上述权限清单 → 订阅 `im.message.receive_v1` / `card.action.trigger` / `p2p_chat_create_v1` → 打开「长连接模式」→ 发布到测试企业（**无需审核**）
3. **30 分钟**：`pnpm add @larksuiteoapi/node-sdk` → 起 `WSClient` → 测试群拉机器人进 → 发消息看控制台 log → 推一张静态卡片回去 ✅

## 十、对开发的直接影响（Action Items）

| 来自这份文档的限制 | 我们要做什么 |
|---|---|
| 不能监听员工私聊 | 产品定位**只做群协作**，文案不要承诺「全公司信息抓取」|
| 群里要先被拉进去 | Demo 脚本第一步就是「群主 @ 拉机器人」，别假装机器人凭空就在 |
| 5 req/秒 频控 | 批处理（30s / 10 条）+ 流式卡片 0.5s 节流 |
| 卡片只能用预置组件 | 原型设计阶段（袁晨杰那条 todo）就用 [飞书卡片搭建工具](https://open.feishu.cn/tool/cardbuilder) 拖，别画 Figma 自由稿，画了实现不了 |
| 但 H5 应用能写完整前端 | 真要 Figma 自由设计的页面，挂成 H5 应用走 webview 加载即可。MVP 不做，决赛知识图谱可视化可以走这条 |
| 机器人菜单常被忘掉 | 开发者后台直接配三个按钮：「📊 今日缺口」「🧠 我的 Memory」「⚙️ 静默 1 小时」，零代码加分项 |
| 多维表格 10 QPS | Memory 写入也要批，别每条 LLM 输出立即落库 |
| 妙记没有 push 事件 | 老板手动 @ 妙记链接给机器人，作为产品流程的一步（也是天然的 demo 触发点）|
| Aily 是竞品 | 决赛 5 分钟里**主动提**一句「为什么不直接用 Aily」，把差异讲清楚 |
