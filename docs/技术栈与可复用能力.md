## 一、我们要造的东西（一句话回顾）

群聊里的隐形同事：监听对话 → 检测信息缺口 → 调飞书检索 API 把答案拉回来 → 用一张可交互卡片推回群里。

不是「被 @ 才动」，而是「在你说话时已经在跑」。

---

## 二、产品在什么基础上 coding

我们不是从零造一个 IM 系统。Sentinel **本身就是一个飞书应用**——和飞书的关系，跟 iOS App 跟 iPhone 一样。

```
┌─────────────────────────────────────────────┐
│  飞书 App（用户在 PC / 手机上用）            │  ← 用户根本不知道我们存在
└──────────────┬──────────────────────────────┘
               │ 用户在群里说话
               ▼
┌─────────────────────────────────────────────┐
│  飞书开放平台 (open.feishu.cn)              │  ← 飞书托管的事件总线
│  事件订阅 / 卡片回调                         │
└──────────────┬──────────────────────────────┘
               │ WebSocket 长连接推事件
               ▼
┌─────────────────────────────────────────────┐
│  我们写的 Sentinel Server (Node + TS)        │  ← 这是要 coding 的部分
│  ├─ 接收事件                                │
│  ├─ 缺口检测（豆包 Lite）                    │
│  ├─ Skill Router → 决定调哪个源              │
│  ├─ 检索（Wiki / Bitable / 群历史 / 妙记）   │
│  ├─ 生成卡片（豆包 Pro）                     │
│  ├─ Memory（SQLite）记录总结事件             │
│  └─ 推回群里                                │
└─────────────────────────────────────────────┘
```

**没有前端**——UI 就是飞书的卡片。10–15 个文件、500–1000 行代码就能跑通 MVP。

## 三、技术栈（已敲定）

| 层 | 选型 | 理由 |
|----|------|------|
| **语言 / 运行时** | TypeScript + Node 20 | 飞书 Node SDK 最完整；卡片 JSON 操作天然顺手；团队上手快 |
| **飞书 SDK** | `@larksuiteoapi/node-sdk` | 官方维护，含 WebSocket 长连接客户端 |
| **接入模式** | WebSocket 长连接 | demo 阶段免公网部署、改完即生效 |
| **消息触发权限** | `im:message`（全量监听群消息） | 直接呈现 Sentinel 完整形态，不走「@ 才触发」的妥协路径 |
| **小模型（缺口检测）** | 豆包 Lite (Doubao-1.5-lite) | 字节官方、中文强、便宜，二分类判断够用 |
| **大模型（检索改写 + 卡片生成）** | 豆包 Pro (Doubao-1.5-pro) | 字节官方、上下文理解强、赛场加分 |
| **成本控制** | 两层模型起步，关键词预过滤 / 滑窗批处理后续再加 | 配额无限，demo 阶段先跑通；上线话术里再讲完整方案 |
| **检索来源** | Wiki + Bitable + 群历史 + 妙记（**全部接权限**） | 由 Skill Router 在运行时决定调哪个 |
| **路由层** | Skill Router（自研） | 大模型读消息上下文 → 输出该调哪几个源 → 并行查 |
| **Memory** | 通用 memory 文档（SQLite 持久化） | 记录每次「总结事件」，下次同话题可直接复用 |
| **存储** | SQLite (better-sqlite3) | 群上下文画像、检索缓存、反馈记录、Memory 总结 |
| **部署** | 纯本地跑 demo | 长连接模式天然支持，零部署成本 |
| **后端框架** | 单进程裸 Node（不上 Nest / Express） | demo 不需要分层 |
| **包管理** | pnpm | 全局共识 |

## 四、飞书生态可复用能力盘点

### 1. 飞书 CLI（脚手架）

- **是什么**：`lark-cli`，可一键 `lark init` 起项目骨架。
- **结论**：**不用**。SDK 自带 example 复制粘贴一样快，CLI 不是命脉。

### 2. 机器人 Server SDK（核心）

- **是什么**：`@larksuiteoapi/node-sdk`，订阅 `im.message.receive_v1` 事件。
- **权限 scope**：
  - `im:message` —— 全量群消息（我们的形态）
  - `im:message.history_v1` —— 拉群历史（检索用）
  - `im:chat:readonly` —— 读群信息
- **接入方式**：WebSocket 长连接（`Client.ws.start()`）。
- **优先级**：**高**。命脉。
- **文档**：<https://open.feishu.cn/document/server-docs/im-v1/message/events/receive>

### 3. 文档 / 多维表格 / 知识库检索（差异化核心）

- 知识库搜索：`/open-apis/wiki/v2/spaces/{id}/nodes/search`
- 多维表格搜索：`/bitable/v1/apps/{app_token}/tables/{table_id}/records/search`
- 妙记会议纪要：`/minutes/v1/...`（具体接口需注册后从开发者后台查）
- **权限**：`docx:document:readonly`、`bitable:app:readonly`、`wiki:wiki:readonly`、`drive:drive:readonly`、`minutes:minutes:readonly`
- **优先级**：**全部高**。我们的 Skill Router 会动态选用。
- **文档**：<https://open.feishu.cn/document/server-docs/docs/wiki-v2/space-node/search>

### 4. 交互卡片（Message Card 2.0）

- JSON Schema 卡片，支持 button / select / form / markdown。
- 按钮回调：`card.action.trigger` 事件，`action.value` 自定义字段 → 承载「分享 / 追问 / 不需要」三按钮。
- **流式更新**：`PATCH /open-apis/im/v1/messages/{id}` → 演示「正在思考…」过渡动画。
- **优先级**：**高**。产品形态本体。
- **文档**：<https://open.feishu.cn/document/uAjLw4CM/ukzMukzMukzM/feishu-cards/card-json-v2-structure>

### 5. 妙记 / Aily / MyAI（竞品 + 数据源）

- **Aily**：飞书官方 AI 应用搭建平台，自带 RAG + 工作流 → **和 Sentinel 高度重叠**。
- **MyAI**：群里官方 AI 助手，**被动问答**（要主动唤起）。
- **妙记**：会议纪要 → **作为 Sentinel 的检索源之一**。
- **差异化叙事**：Aily / MyAI 都需要主动唤起，Sentinel 主动监听 + 缺口检测——这是它们的盲区。
- **优先级**：妙记 = 检索源（高），Aily / MyAI = 竞品对比素材（高）。

### 6. 本地开发体验

- **WebSocket 长连接**：事件直推本地，**无需 ngrok / 公网部署**。
- **测试企业 + 版本管理**：发布「测试版本」只对自己生效，不污染正式应用。
- **事件回调调试**：开放平台后台可手工触发事件 payload。
- **文档**：<https://open.feishu.cn/document/server-docs/event-subscription-guide/event-subscription-configure-/long-connection-mode>

## 五、SDK 核心接口速查（写代码会用到的部分）

```typescript
// ① 初始化
const client = new lark.Client({ appId, appSecret });

// ② WebSocket 长连接接收事件
const wsClient = new lark.WSClient({ appId, appSecret });
wsClient.start({
  eventDispatcher: new lark.EventDispatcher({}).register({
    'im.message.receive_v1': async (data) => { /* 收到群消息 */ },
    'card.action.trigger':    async (data) => { /* 卡片按钮被点 */ }
  })
});

// ③ 发卡片消息
await client.im.message.create({
  params: { receive_id_type: 'chat_id' },
  data: { receive_id, msg_type: 'interactive', content: JSON.stringify(cardJson) }
});

// ④ 流式更新卡片（"正在思考…" → "查到了！"）
await client.im.message.patch({
  path: { message_id },
  data: { content: JSON.stringify(updatedCardJson) }
});

// ⑤ 检索 Wiki / Bitable
await client.wiki.v2.spaceNode.search({ /* keyword */ });
await client.bitable.v1.appTableRecord.search({ /* filter */ });
```

**就这几个接口能搭出 Sentinel 的 80%。**

## 七、Skill Router + Memory 设计要点

> 这是你提出的关键差异化设计，单独拎出来。

### Skill Router

不是写死 if-else，而是**让大模型读 5–10 条上下文消息后输出一个 JSON**：

```json
{
  "should_act": true,
  "intent": "查上次用户调研转化率",
  "sources": ["wiki", "bitable"],
  "reasoning": "提到了报表数据，可能在多维表格；也可能在调研文档里"
}
```

主程序拿到这个 JSON → **并行**调用对应的检索接口 → 把结果再喂大模型生成卡片。

**好处**：
- 加新数据源只需在 Router prompt 里加一行说明
- 评委问「为什么调这个源」时，`reasoning` 字段就是答案
- 比硬编码路由灵活 10 倍

### Memory（通用总结文档）

每次 Sentinel 输出一张卡片后，把「触发上下文 + 检索结果 + 用户反馈（点了哪个按钮）」存进 SQLite：

```sql
CREATE TABLE memory (
  id INTEGER PRIMARY KEY,
  chat_id TEXT,
  trigger_context TEXT,    -- 触发的对话片段
  intent TEXT,             -- Router 识别的意图
  sources_used TEXT,       -- 调了哪些源
  result_summary TEXT,     -- 检索 + 整合后的总结
  user_feedback TEXT,      -- shared / dismissed / asked_more
  created_at INTEGER
);
```

下次同一群再聊到相似话题，先查 Memory → 命中就直接复用，**省一次 LLM + 一次检索**，还能讲「越用越懂你们群」的故事。

## 八、申请 + 跑通流程（一次性步骤）

### A. 注册 + 建应用（30 分钟）

1. 飞书账号 → 创建「测试企业」（免费、3 人内不要钱）
2. 开发者后台 <https://open.feishu.cn/app> → 创建「企业自建应用」
3. 拿到 **App ID** + **App Secret**

### B. 开权限 + 订阅事件（10 分钟）

4. 开权限：`im:message`、`im:message.history_v1`、`im:chat:readonly`、`wiki:wiki:readonly`、`bitable:app:readonly`、`docx:document:readonly`、`drive:drive:readonly`、`minutes:minutes:readonly`
5. 订阅事件：`im.message.receive_v1`、`card.action.trigger`
6. **打开「长连接模式」开关**
7. 「发布到测试企业」（一键，无审核）

### C. 本地跑通（30 分钟）

8. `pnpm init && pnpm add @larksuiteoapi/node-sdk dotenv better-sqlite3`
9. App ID / Secret 写进 `.env`
10. 写 10 行代码启动 `WSClient`
11. 测试企业建群，拉机器人进群
12. 群里发消息 → 控制台打印 → 链路通了

### D. 集成测试方式

| 测什么 | 怎么测 |
|------|------|
| 接收消息 | 测试群手发消息，看控制台 log |
| 卡片渲染 | 推卡片 JSON，肉眼看 + 飞书后台「卡片搭建工具」可视化预览 |
| 按钮回调 | 群里点按钮，看本地是否收到 `card.action.trigger` |
| 检索 API | 测试企业建几个文档/多维表格作为「假数据」，调 search API |
| 端到端 | 团队成员在测试群「演戏」对话，看 Sentinel 反应 |

**不需要**：域名 / 服务器 / 应用市场审核 / 付费。
