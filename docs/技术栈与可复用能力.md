>

---

## 一、产品一句话

群聊里的隐形同事：监听对话 → 检测信息缺口 → 调飞书检索 API 拉答案 → 推一张可交互卡片回群。

**不是「被 @ 才动」，而是「在你说话时已经在跑」。**

## 二、技术栈

| 层 | 选型 |
|----|------|
| 语言 / 运行时 | TypeScript + Node 20 |
| 飞书 SDK | `@larksuiteoapi/node-sdk` |
| 接入模式 | WebSocket 长连接（免公网部署） |
| 监听权限 | `im:message`（全量群消息） |

| **存储（语义）** | **Chroma** —— 只存向量 + id，原文留在多维表格 |
| 后端框架 | 单进程裸 Node |
| 部署 | 纯本地跑 demo |
| 包管理 | pnpm |

## 四、存储层方案

**多维表格 + Chroma 双库分工**：

| 存什么 | 放哪 |
|------|------|
| Memory 总结、阶段快照、决策、待办、反馈 | 多维表格（结构化 + 可视化） |
| 知识图谱关系（决策→数据→负责人） | 多维表格双向关联字段（轻量图谱，1-2 跳够 demo） |
| Memory summary 的向量 | Chroma（用同一个 id 主键关联多维表格那边） |

**为什么不用 SQLite**：飞书多维表格自带 UI，演示时直接打开给评委看「这就是知识图谱」，比讲表 join 强 10 倍；算深度集成飞书原生能力，加分实打实。

**为什么 Chroma 必须有**：「那个数据」要找到「用户转化率指标」这种语义匹配，多维表格做不了，必须向量库。Chroma 只做"语义索引"这一件事，不存原文，不会和多维表格抢职责。

**风险与边界**：3+ 跳图遍历（最短路径、社区发现）做不了——但 demo 用不到。Phase 2 上火山引擎 veDB Graph。

**调研结论**：飞书开放平台**没有**通用知识图谱 API、没有 TSDB，`lark-cli` 也只是脚手架不暴露业务能力。火山引擎有 veDB Graph + TSDB 但不在飞书 SDK 体系内，MVP 不引入。

## 五、SDK 核心接口（写代码会用到的部分）

```typescript
// ① 初始化
const client = new lark.Client({ appId, appSecret });

// ② WebSocket 长连接接收事件
const wsClient = new lark.WSClient({ appId, appSecret });
wsClient.start({
  eventDispatcher: new lark.EventDispatcher({}).register({
    'im.message.receive_v1': async (data) => { /* 收到群消息 */ },
    'card.action.trigger':    async (data) => { /* 卡片按钮被点 */ }
  })
});

// ③ 发卡片
await client.im.message.create({
  params: { receive_id_type: 'chat_id' },
  data: { receive_id, msg_type: 'interactive', content: JSON.stringify(cardJson) }
});

// ④ 流式更新卡片（"正在思考…" → "查到了！"）
await client.im.message.patch({ path: { message_id }, data: { content: ... } });

// ⑤ 检索
await client.wiki.v2.spaceNode.search({ ... });
await client.bitable.v1.appTableRecord.search({ ... });
```

**就这几个接口能搭出 Sentinel 的 80%。**

## 六、Skill Router 设计

不写死 if-else，让大模型读 5–10 条上下文输出 JSON：

```json
{
  "should_act": true,
  "intent": "查上次用户调研转化率",
  "sources": ["wiki", "bitable"],
  "reasoning": "提到了报表数据，可能在多维表格；也可能在调研文档里"
}
```

主程序拿到 JSON → 并行调对应检索接口 → 结果喂大模型生成卡片。

**好处**：加新数据源只需改 prompt；评委问「为什么调这个源」时 `reasoning` 就是答案。

## 七、批量处理策略（沛彤 issue #5 沿用）

- 默认每 30 秒或攒 10 条消息一批送 LLM
- **强信号词立即触发**（"定了""行动项""就这样"），不等批次，避免决策延迟

数据 schema 要素（采纳沛彤设计）：滑动窗口 20 条上下文 + `reply_to` 话题追踪 + 4 种信息类型（gap/decision/action/risk）+ 阶段快照「追加不删」。

## 八、申请 + 跑通流程

**A. 注册（30 分钟）**：飞书账号 → 创建测试企业 → 开发者后台建「企业自建应用」→ 拿 App ID + Secret

**B. 配权限（10 分钟）**：勾权限 `im:message` / `im:message.history_v1` / `wiki:wiki:readonly` / `bitable:app:readonly` / `docx:document:readonly` / `minutes:minutes:readonly` → 订阅事件 `im.message.receive_v1` / `card.action.trigger` → 打开**长连接模式** → 发布到测试企业（无审核）

**C. 跑通（30 分钟）**：

```bash
pnpm init
pnpm add @larksuiteoapi/node-sdk dotenv chromadb
```

写 10 行启 `WSClient` → 测试群拉机器人进 → 发消息看控制台 log → 推一张静态卡片回去。

**不需要**：域名 / 服务器 / 应用市场审核 / 付费。
