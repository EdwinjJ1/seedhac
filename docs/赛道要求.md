# 赛道要求：基于 IM 的办公协同智能助手（公开版）

> **命题**：Agent-Pilot · 从 IM 对话到演示稿的一键智能闭环

---

## 一、考察重点（评审视角）

| 维度 | 描述 | 对应我们的关键决策 |
|------|------|--------------------|
| AI Native 思维 | Agent 驱动核心流程，而非堆砌 GUI | 被动监听 + Skill Router，不依赖按钮触发 |
| 多端工程能力 | iOS/Android + macOS/Windows 实时双向同步 | 飞书 WebSocket 长连接 + Card 2.0 多端一致渲染 |
| 端到端交付 | 高质量、高可用的复杂应用 | monorepo contracts-first 架构 + Result<T> 错误处理 |

**核心理念（评审会反复问）**：AI Agent 是**主驾驶（Pilot）**，GUI 是**仪表盘与辅助操作台（Co-pilot）**。

---

## 二、必须完成项 Must-have

### M1 · 多端协同框架

- [ ] 移动端（iOS 或 Android）可操作
- [ ] 桌面端（macOS 或 Windows）可操作
- [ ] 一端操作实时反映到另一端（双向同步）

**在本项目中的对应**：飞书 Card 2.0 天然多端，状态同步通过 Bitable + 消息回调实现。需要验证 Card 按钮点击在手机和电脑上均能触发 callback 并刷新。

---

### M2 · 六大场景（必须覆盖，可自由编排）

| 场景 | 名称 | 强制要求 | 项目中对应的 Skill |
|------|------|----------|-------------------|
| **A** | 意图/指令入口 | 文本**或**语音，群聊**或**单聊 | 所有 Skill 的 `match()` + 被动监听逻辑 |
| **B** | 任务理解与规划 | LLM/Planner 拆解子任务 | `skill-router.ts` → Skill 编排 |
| **C** | 文档/白板生成与编辑 | 自动生成并可迭代 | `summary.ts` → `docx-client.ts` 写飞书文档 |
| **D** | 演示稿/自由画布生成与排练 | PPT **或**自由画布，支持修改 | `slides.ts` → 飞书 SML 2.0 生成 PPT |
| **E** | 多端协作与一致性 | 跨端状态一致 + 冲突解决 | Bitable 作为单一状态源，Card callback 幂等 |
| **F** | 总结与交付 | 分享链接**或**导出文件 | `archive.ts` → Bitable 归档 + 分享链接生成 |

**约束红线**：
- 必须覆盖 **IM + Doc + PPT/画布** 三类工具（缺一不可）
- AI Agent 必须是主流程驱动者，GUI 只做确认和微调
- 各场景可单独 demo，但**至少一次多场景组合编排**

## 三、加分项 Good-to-have

| 加分项 | 实现思路 | 优先级 |
|--------|---------|--------|
| 离线支持 | 本地缓存 + 恢复后 diff 合并 | 低（工程复杂，demo 难展示） |

| 富媒体与画布操作 | 在文档/画布中插入图片/表格 | 中 |

## 四、演示验收 Checklist

最终 demo 需清晰展示以下三点（缺一会被扣分）：

### Demo 必演场景

- [ ] **多端协同**：手机操作 → 电脑实时刷新（或反向），展示双向同步
- [ ] **Agent 驱动主流程**：IM 群聊/单聊输入自然语言 → Agent 自动编排多个场景模块（至少串联 2 个场景），全程不需要手动切换应用
- [ ] **Office 套件覆盖**：单次 demo 中同时体现 IM 消息、飞书文档（或白板）、飞书 PPT 三者的联动

### 建议的 Demo 黄金路径

```
用户在群聊输入："帮我把上周讨论的需求整理成 PPT，明天要汇报"
        ↓ (场景 A) 意图捕捉
Agent 识别意图 → 启动编排
        ↓ (场景 B) 任务规划
Planner 拆解：检索历史讨论 → 生成文档 → 生成 PPT → 归档
        ↓ (场景 C) 文档生成
自动创建飞书文档，写入结构化需求摘要（多端可见）
        ↓ (场景 D) PPT 生成
基于文档内容生成演示稿，发送预览卡片到群里
        ↓ (场景 E) 多端同步
手机端点击"修改标题" → 桌面端文档实时更新
        ↓ (场景 F) 交付
生成分享链接 + 写入 Bitable 归档，回复汇报摘要卡片
```

## 五、评审容易追问的问题

1. **为什么用 Agent 而不是写死流程？** → 场景可独立编排，用户意图变化时不需要改代码
2. **多端同步冲突怎么处理？** → Bitable 作为单一状态源，last-write-wins + 消息 callback 幂等
3. **被动监听如何避免打扰用户？** → 小模型先做 gap detection，只有信息缺口明显才触发；QA 需要显式 @
4. **和现有飞书机器人的区别？** → 不需要 @，持续理解上下文，输出结构化交付物而非回复文本

## 六、需求与代码的快速索引

```
场景 A (意图入口)     → packages/bot/index.ts        (WebSocket 事件接收)
场景 B (任务规划)     → packages/bot/skill-router.ts  (Router + LLM 编排)
场景 C (文档生成)     → packages/skills/summary.ts    + packages/bot/docx-client.ts
场景 D (PPT 生成)     → packages/skills/slides.ts     (飞书 SML 2.0)
场景 E (多端同步)     → packages/bot/bitable-client.ts (状态存储)
                        + packages/bot/card-builder.ts  (Card callback 刷新)
场景 F (归档交付)     → packages/skills/archive.ts    + Bitable 知识图谱
```
